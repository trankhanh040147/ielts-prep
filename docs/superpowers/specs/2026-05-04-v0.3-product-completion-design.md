## Overview

Finish v0.3 by completing the remaining product-focused roadmap items while keeping the app local-first and Task 2-only.

This pass does not include Task 1 support, cloud sync, auth, flashcards, or tooltip explanations for individual diff tokens. Those remain future work.

The current app already has the main topic picker, AI topic generation, editable topic names, history rename/delete, feedback diff view, and localStorage persistence. v0.3 completion builds on those pieces with minimal new architecture.

---

## Goals

- Let users paste or type their own IELTS Writing Task 2 prompt.
- Show a live word count for the current draft.
- Return and display an estimated IELTS band breakdown for each feedback request.
- Save and reload band estimates with each practice record.
- Make local history useful for tracking multiple attempts on the same prompt.

## Non-Goals

- No user accounts, cloud sync, or backend database.
- No Task 1 graph/chart/map/process workflows.
- No separate scoring-only workflow.
- No local heuristic band scoring; the estimate comes from Gemini and is normalized defensively.
- No migration of the localStorage key; existing records should continue loading with optional new fields missing.

## Data Model

Add a band estimate type to the frontend and backend shared type files.

```ts
type BandEstimate = {
  overall: number
  taskAchievement: number
  coherenceCohesion: number
  lexicalResource: number
  grammaticalRangeAccuracy: number
  summary: string
}
```

Rules:
- Scores are numbers from `0` to `9`, allowing half-band decimals like `6.5`.
- Numeric values outside `0..9` are clamped by the mapper rather than crashing the request.
- If Gemini omits the whole estimate or returns invalid scoring, the app still displays normal feedback without a score.

Extend `PracticeRecord`:

```ts
type PracticeRecord = {
  id: string
  mode: PracticeMode
  prompt: string
  topicName: string
  draft: string
  feedback: FeedbackUnit[]
  bandEstimate?: BandEstimate
  updatedAt: string
}
```

No explicit `attemptNumber` is persisted. Attempt ordering is derived from records that share the same `mode` and `prompt`, sorted newest first by `updatedAt`. This avoids stale attempt numbers after deletion or rename.

## Backend Design

### Feedback API Response

Keep the existing endpoint:

```http
POST /api/feedback
```

Extend the response shape from:

```json
{ "feedback": [] }
```

to:

```json
{
  "feedback": [],
  "bandEstimate": {
    "overall": 6.5,
    "taskAchievement": 6.5,
    "coherenceCohesion": 6,
    "lexicalResource": 6.5,
    "grammaticalRangeAccuracy": 6,
    "summary": "Clear position, but development and grammar need more precision."
  }
}
```

`bandEstimate` is optional. Existing frontend code should not break if it is absent.

### Gemini Prompt

Update `getGeminiFeedback()` so the Gemini instruction requests both feedback and scoring in one JSON payload.

Required schema:

```json
{
  "feedback": [
    {
      "targetText": "string",
      "strengths": ["string"],
      "issues": ["string"],
      "revision": {
        "explanation": "string",
        "rewrites": ["string"]
      }
    }
  ],
  "bandEstimate": {
    "overall": 6.5,
    "taskAchievement": 6.5,
    "coherenceCohesion": 6,
    "lexicalResource": 6.5,
    "grammaticalRangeAccuracy": 6,
    "summary": "string"
  }
}
```

The scoring instruction should say the estimate is approximate and should use IELTS Writing Task 2 criteria:
- Task Achievement
- Coherence and Cohesion
- Lexical Resource
- Grammatical Range and Accuracy

### Mapping and Validation

Keep `mapGeminiToFeedback()` for feedback units. Add a focused score mapper, for example `mapGeminiToBandEstimate(raw: unknown): BandEstimate | undefined`.

Validation behavior:
- Return `undefined` if `raw.bandEstimate` is missing or not an object.
- Accept numeric strings only if they parse cleanly to finite numbers.
- Round scores to the nearest `0.5`.
- Clamp scores to `0..9`.
- Return `undefined` if any of the five required score fields is missing or cannot be parsed as a finite number.
- Require a non-empty `summary`; if missing, use `Estimated IELTS band based on this draft.`.

`feedbackRoute` should return both mapped values:

```ts
return res.json({ feedback, bandEstimate })
```

## Frontend Design

### Custom Topic Prompt

Extend `TopicPicker` with a custom prompt area.

UI behavior:
- Add a secondary action labelled `Use custom topic`.
- Clicking it shows a textarea labelled `Custom IELTS Task 2 topic`.
- Typing in the textarea updates the current `prompt` in `App.tsx`.
- The session name auto-fills from the first 5 meaningful words of the custom prompt when the custom prompt is first entered.
- After the user manually edits the session name, further custom prompt typing should not overwrite that custom name.
- If the custom prompt is cleared, keep the current custom topic mode visible and leave `PromptCard` showing an empty or helper state rather than silently switching topics.

Minimal helper:

```ts
function deriveTopicName(prompt: string): string {
  const words = prompt.trim().split(/\s+/).filter(Boolean).slice(0, 5)
  return words.length === 0 ? 'Custom Topic' : words.join(' ')
}
```

### Word Count

Add word count display to `DraftEditor`.

Counting rule:
- Split on whitespace after trimming.
- Empty or whitespace-only drafts count as `0`.
- Punctuation stays attached to words; `education.` counts as one word.

Display:
- Show `0 words`, `1 word`, or `N words` below the textarea.
- Keep it informational only; no validation threshold in v0.3.

### Band Estimate Display

Add a focused `BandEstimatePanel` component and render it at the top of `FeedbackPanel`.

Display:
- Overall band appears first and prominent.
- Four criteria appear as compact rows or chips.
- Summary appears as a short note below the scores.
- If no `bandEstimate` exists, render nothing.

Suggested labels:
- Overall
- Task Achievement
- Coherence & Cohesion
- Lexical Resource
- Grammar Range & Accuracy

### App State and Feedback Flow

Add state:

```ts
const [bandEstimate, setBandEstimate] = useState<BandEstimate | undefined>()
```

On successful feedback response:
- Set `feedback` from `result.feedback ?? []`.
- Set `bandEstimate` from `result.bandEstimate`.

On mode change:
- Reset draft, feedback, feedback error, band estimate, and session id.

On selecting history:
- Restore `bandEstimate` from the record.

On saving:
- Include `bandEstimate` in the saved record if available.

### Feedback History Per Prompt

Keep the current flat localStorage array. Improve display so multiple attempts are understandable.

For each history row, show:
- Topic name
- Mode badge
- Date
- Word count for the saved draft
- Overall band if available
- Attempt label derived from prompt grouping: `Attempt 1`, `Attempt 2`, and so on, where `Attempt 1` is the oldest saved record for that same `mode + prompt`.

This gives improvement tracking without adding a new nested storage model.

## Error Handling

| Case | Handling |
|---|---|
| Custom prompt is blank | `handleCheck` sets `feedbackError` to `Topic prompt is required.` and does not call the API. |
| Gemini returns feedback but no band estimate | Show feedback normally and omit score panel. |
| Gemini returns invalid band fields | Normalize valid fields; if no valid object remains, omit score panel. |
| Old localStorage records have no `bandEstimate` | Load normally; history row just has no band label. |
| User saves draft without feedback | Save as today; record has no band estimate. |
| API failure | Preserve draft, prompt, topic name, current feedback, and current band estimate. Show existing retry banner. |
